# Big Shop

Recipe management and shared shopping-list generation: users collect Recipes, select which to cook, and get an aggregated Shopping List of what to buy.

## Language

**Account**:
The sharing/ownership boundary. A group of one or more Users (invited via email) who see and edit the same Recipes and Shopping List. All recipe and shopping-list data is scoped to an Account, never to an individual User directly.
_Avoid_: Household, User (when meaning the shared group), Family

**User**:
An individual authenticated identity (Auth0). Distinct from Account: the User is who's logged in, the Account is what data they see. **Known limitation, not a permanent rule:** every code path today (accepting an Invite disables all of that User's other memberships; `account_user` lookups assume one enabled row) treats a User as belonging to at most one Account at a time. Genuine multi-Account membership (e.g. one person sharing two different households' lists) is a legitimate future direction, not ruled out — the current single-membership behavior just hasn't been built past yet.

**Invite**:
A pending offer for a specific email address to join a specific Account, created by an existing member and accepted or rejected by the invitee. Accepting one moves the invitee's active membership to that Account (see the known limitation under User).

**Global Catalog** (Ingredient, Unit, Tag, Department):
Ingredient, Unit, Tag, and Department are shared across every Account rather than scoped to one — there is exactly one "gram" and one "tomato" in the whole system, not one per Account. This is deliberate (see [ADR-0001](./docs/adr/0001-global-ingredient-catalog.md)): it's what lets ingredient/unit metadata (e.g. average weight, preferred unit) accumulate value from every Account's usage rather than being re-entered per Account. Two different extensibility rules apply within it:
- **Open** (Ingredient, Unit): any User can introduce a new one just by typing it when saving a Recipe — no curation step.
- **Closed** (Tag, Department): a fixed, curated set seeded once by migration. No code path exists to add a new Tag or Department at runtime today.

**Shopping List**:
The single active, mutable set of Shopping List Items for an Account — generated by selecting Recipes to cook, then shopped from and cleared. Not versioned or historical; see Shopping List Event for the historical record of what happened to it over time.

**Generate Shopping List**:
The action of (re)computing every Ingredient Item from a given set of Recipes. It's a full replace, not an add — every existing Ingredient Item is discarded and recreated from the complete Recipe set passed in; Extra Items are untouched either way. The Shopping List page's Recipe checkboxes naturally supply the complete set each time; Dave's chat tool is delta-oriented ("add this recipe"), so it reads the current Shopping List first and unions in the new Recipe before calling the same full-replace action.

**Shopping List Item**:
One line on the Shopping List, identified by its display name alone — an Ingredient Item and an Extra Item that happen to share a name are the same line, one bought/unbought checkbox, account-wide. Two kinds:
- **Ingredient Item**: auto-generated by aggregating quantities for a given Ingredient across every selected Recipe. Carries a real quantity, Unit, and Department.
- **Extra Item**: added directly by a User, not derived from any Recipe. Carries no meaningful quantity or Unit — it's a plain checklist entry (a name and a bought/unbought state), even though the underlying row stores placeholder quantity/unit values.
_Avoid_: List Item (ambiguous with Extra Item specifically — always say which kind, or "Shopping List Item" for the general concept)

**Recipe**:
A named dish a User has added to their Account, with Ingredient Lines, Method (instructions), and Tags. Belongs to an Account, not to an individual User.

**Tag**:
A closed-vocabulary label attached to a Recipe (e.g. "Vegetarian", "Batch Cook") for filtering/browsing. Not user-extensible — see Global Catalog.

**Ingredient**:
A canonical, Global Catalog entry for a foodstuff (e.g. "tomato") — just an identity, no quantity. What a Shopping List is ultimately built out of.
_Avoid_: Using "Ingredient" for a Recipe's specific quantity of one — that's an Ingredient Line.

**Ingredient Line**:
One row of a Recipe's ingredient list: a quantity + Unit of a specific Ingredient, scoped to that one Recipe (DB table `part`). Distinct from Ingredient itself — many Ingredient Lines across many Recipes can reference the same Ingredient.
_Avoid_: Ingredient (when meaning this), Part, Recipe Ingredient

**Unit**:
A Global Catalog measurement (e.g. "gram", "teaspoon", "bunch") that quantifies an Ingredient Line or Shopping List Item. Free-text extensible today — a User can introduce a never-seen-before Unit just by typing it when saving a Recipe.

**Department**:
The aisle-style grouping a Shopping List Item is shown under (e.g. "vegetables", "meat and fish", "other"). One per Ingredient (many-to-one), not many-to-many — the join table (`ingredient_department`) is looser than this rule today, which is a defect (an Ingredient with two Department rows would silently duplicate that Ingredient Line wherever it's read), not an intended feature.

**Recipe Import**:
Populating a new Recipe's name/Ingredient Lines/Method/vegetarian Tag from an external source instead of typing every field by hand. Chosen via one of three Import Sources on the New Recipe page.
_Avoid_: Scraping (see URL Import — the old per-site scraper approach this replaced)

**Import Source** — URL Import, Photo Import, Manual Entry:
- **URL Import**: user supplies a recipe's web address; the server fetches and strips its raw HTML and makes one LLM call to extract name/ingredients/method/vegetarian-ness. Works against any recipe site (`pages/api/parse-recipe-url.js`) — this fully replaced the older approach of a hand-written DOM-selector-plus-regex scraper per named site (Simply Recipes, BBC Good Food, etc., formerly under `pages/api/third-parties/`, now deleted).
- **Photo Import**: user uploads a photo of a recipe; GPT-4 Vision extracts it as a Processing Job, since the extraction call runs too long for a synchronous Lambda response.
- **Manual Entry**: user fills fields directly — but can still paste freeform multiline ingredient text into a bulk box that gets LLM-parsed into structured Ingredient Lines, rather than typing one row per ingredient.
_Avoid_: Camera (UI label for Photo Import's tab; "Photo Import" is the canonical term)

**Processing Job**:
The async unit of work behind Photo Import specifically (not used by URL Import or Manual Entry, which respond synchronously). The client polls for completion rather than waiting on one request, because Netlify Functions' Lambda-based execution can't hold a request open long enough for a Vision call to finish.

### Meal planning intelligence (experimental, not fully productionized)

The terms below exist to support Dave and are newer / less battle-tested than the rest of this glossary — treat them as more likely to change shape than e.g. Account or Recipe.

**Dave**:
The conversational meal-planning assistant (GPT-3.5-turbo, tool-calling). Only ever searches and acts within the calling User's own Account — never the Global Catalog or other Accounts' Recipes.

**Shopping List Event**:
An append-only log entry recording something that happened to the Shopping List (a Recipe added, an Extra Item added/removed, the list cleared) — the raw material Dave uses to infer Recent Recipe / Favorite Recipe. Not a snapshot of list contents at a point in time.
_Avoid_: Shopping List History (as a table/entity name — there is no such table; "history" is the informal name for the feature built on Shopping List Event, used in the `/shopping-list/history` endpoint and CLAUDE.md's docs, which currently misnames the underlying table)

**Recent Recipe**:
A Recipe added to the Shopping List within a recent trailing window of time, surfaced to Dave for meal-planning suggestions.

**Favorite Recipe**:
A Recipe added to the Shopping List on more than one distinct calendar day (not just counted more than once — regenerating the list twice in one day only counts as one usage). Distinguishes genuine repeat cooking from bulk list-regeneration noise.
